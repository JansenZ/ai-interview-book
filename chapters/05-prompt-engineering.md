# 第五章：Prompt Engineering 入门

> 💡 目标：写出高质量的提示词，让 AI 输出你想要的结果
> 
> ⭐⭐⭐ 面试必考 + 实际工作必会

---

## 📖 什么是 Prompt Engineering？

**定义**：设计和优化提示词（Prompt），让大模型输出更好的结果。

**类比**：
```
传统编程：写代码告诉计算机做什么
Prompt Engineering：写文字告诉 AI 做什么
```

**为什么重要**：
- 同样的模型，不同的 prompt 效果天差地别
- 好的 prompt 可以激发模型的"涌现能力"
- 是前端开发最常用的 AI 技能

---

## 🎯 好 Prompt vs 坏 Prompt

### 坏 Prompt 示例

```
❌ "写个函数"
❌ "总结一下"
❌ "这个代码有什么问题？"
```

**问题**：
- 太模糊，AI 不知道你想要什么
- 没有上下文，AI 只能瞎猜
- 没有格式要求，输出难以使用

### 好 Prompt 示例

```
✅ "用 JavaScript 写一个防抖函数，要求：
   1. 接收函数和延迟时间两个参数
   2. 返回一个新的函数
   3. 在延迟时间内只执行一次
   4. 添加 TypeScript 类型定义
   5. 包含使用示例"

✅ "总结下面这篇文章的核心观点，要求：
   - 不超过 200 字
   - 分 3 个要点
   - 用 bullet points 格式
   
   [文章内容]"

✅ "审查下面这段代码，指出：
   1. 潜在的性能问题
   2. 可能的安全漏洞
   3. 代码风格问题
   4. 给出改进建议
   
   [代码内容]"
```

**好的地方**：
- 明确的任务描述
- 具体的要求（格式、长度等）
- 提供必要的上下文

## 🏗️ Prompt 的核心结构

### 基础公式

```
好 Prompt = 角色 + 任务 + 上下文 + 要求 + 示例（可选）
```

### 1. 角色（Role）

**作用**：设定 AI 的身份，让它用特定视角思考

```
"你是一位资深的前端工程师，有 10 年经验..."
"你是一个专业的技术文档写手..."
"你是 Google 的面试官，正在考察候选人..."
```

**效果对比**：
```
❌ "解释一下 React 的 useEffect"

✅ "你是一位 React 专家，正在给初级前端讲解 useEffect。
   请用简单易懂的语言，配合生活化的类比，
   让完全没有 React 经验的人也能理解。"
```

### 2. 任务（Task）

**作用**：明确告诉 AI 要做什么

**技巧**：
- 用动词开头（写、分析、总结、翻译...）
- 任务要具体，不要模糊

```
❌ "处理一下这个数据"

✅ "把这个 JSON 数组转换成表格格式，
   包含姓名、年龄、城市三列"
```

### 3. 上下文（Context）

**作用**：提供背景信息，让 AI 更准确理解

```
"我正在开发一个电商网站的前端，
目标用户是 30-40 岁的女性，
现在需要设计一个商品详情页..."

"这段代码是我们项目的核心模块，
每天处理 100 万 + 请求，
所以性能非常关键..."
```

### 4. 要求（Constraints）

**作用**：限制输出范围，确保结果可用

**常见要求**：
```
- 格式：JSON、Markdown、表格、代码...
- 长度：不超过 200 字、分 3 点...
- 风格：正式、口语化、技术文档...
- 内容：必须包含 X、不能包含 Y...
```

**示例**：
```
"输出要求：
- 用 Markdown 格式
- 分 3 个要点，每个要点不超过 50 字
- 包含代码示例
- 用中文回答"
```

### 5. 示例（Examples）

**作用**：给 AI 展示你想要的输出格式

```
"按照下面的格式输出：

输入：[1, 2, 3, 4, 5]
输出：
- 最小值：1
- 最大值：5
- 平均值：3

输入：[10, 20, 30]
输出：
- 最小值：10
- 最大值：30
- 平均值：20

输入：[100, 200, 300, 400]
输出：
"
```

## 🔧 常用 Prompt 技巧

### 1. Zero-Shot（零样本）

**定义**：不给示例，直接让 AI 做

```
"把这句话翻译成英文：你好，世界"
```

**适用场景**：简单任务，模型本身就会

### 2. Few-Shot（少样本）

**定义**：给几个示例，让 AI 模仿

```
"按照示例转换格式：

输入：2024-01-15
输出：2024 年 1 月 15 日

输入：2024-02-20
输出：2024 年 2 月 20 日

输入：2024-03-25
输出：
"
```

**适用场景**：
- 特殊格式转换
- 特定风格输出
- 模型可能不理解的任务

### 3. Chain of Thought（思维链）⭐⭐⭐

**定义**：让 AI 一步步思考，展示推理过程

**关键句式**：
```
"请一步步思考，展示你的推理过程"
"Let's think step by step"
"先分析问题，再给出答案"
```

**效果对比**：
```
❌ 普通 Prompt：
"小明有 5 个苹果，给了小红 2 个，又买了 3 个，现在有几个？"
AI 可能直接答错

✅ 思维链 Prompt：
"小明有 5 个苹果，给了小红 2 个，又买了 3 个，现在有几个？
请一步步思考，展示计算过程。"

AI 回答：
1. 小明开始有 5 个苹果
2. 给了小红 2 个：5 - 2 = 3 个
3. 又买了 3 个：3 + 3 = 6 个
4. 答案：6 个苹果
```

**面试考点**：
> **Q: Chain of Thought 是什么？为什么有效？**
> 
> **A**: 
> 让 AI 一步步展示推理过程，而不是直接给答案。
> 
> 有效原因：
> 1. 强迫模型分解复杂问题
> 2. 中间步骤可以自我检查
> 3. 减少跳跃性错误
> 
> 实验证明，CoT 可以显著提升数学和推理题的准确率。

### 4. Role Playing（角色扮演）

**定义**：给 AI 设定特定角色

```
"你是一位严厉的代码审查员，专门找代码的问题。
请用挑剔的眼光审查下面这段代码，
指出所有潜在问题，不要留情面。"

"你是一位耐心的编程老师，正在教零基础的学生。
请用简单易懂的语言解释什么是递归，
多用类比，少用术语。"
```

### 5. Output Formatting（格式化输出）

**定义**：明确要求输出格式

```
"请用 JSON 格式输出，包含以下字段：
{
  "summary": "文章摘要",
  "keyPoints": ["要点 1", "要点 2", "要点 3"],
  "sentiment": "positive/negative/neutral"
}"

"请用 Markdown 表格格式输出，包含列：
| 函数名 | 参数 | 返回值 | 说明 |"
```

## 📝 前端开发常用 Prompt 模板

### 1. 代码生成

```markdown
# Role
你是一位资深的前端工程师，擅长 React 和 TypeScript

# Task
编写一个 [组件名称] 组件

# Requirements
- 使用 React + TypeScript
- 使用 Hooks（不使用 class）
- 包含完整的类型定义
- 包含使用示例
- 添加必要的注释

# Features
组件需要支持以下功能：
1. [功能 1]
2. [功能 2]
3. [功能 3]

# Output
请输出完整的代码文件
```

### 2. 代码审查

```markdown
# Role
你是一位经验丰富的代码审查员

# Task
审查下面的代码

# Focus Areas
1. 性能问题（渲染优化、内存泄漏等）
2. 安全问题（XSS、CSRF 等）
3. 代码质量（可读性、可维护性）
4. 最佳实践（React 规范、TypeScript 使用）

# Output Format
对每个问题，请按以下格式输出：
- **问题**：[描述]
- **位置**：[行号]
- **风险等级**：高/中/低
- **建议**：[改进方案]
- **示例代码**：[改进后的代码]

# Code
[粘贴代码]
```

### 3. 技术文档写作

```markdown
# Role
你是一位专业的技术文档写手

# Task
为 [功能/组件] 编写使用文档

# Audience
目标读者：[初级/中级/高级] 前端开发者

# Requirements
- 结构清晰，包含目录
- 包含安装/使用方法
- 包含 API 文档
- 包含至少 3 个使用示例
- 包含常见问题解答

# Output Format
使用 Markdown 格式，遵循以下结构：
1. 简介
2. 安装
3. 快速开始
4. API 参考
5. 示例
6. FAQ
```

### 4. Bug 调试

```markdown
# Context
我在开发一个 [项目类型]，遇到了一个 bug

# Problem
[详细描述问题现象]

# Expected Behavior
[期望的行为]

# Actual Behavior
[实际的行为]

# Code
[相关代码]

# Error Message
[错误信息]

# What I've Tried
我已经尝试过：
1. [尝试 1]
2. [尝试 2]

# Task
请分析可能的原因，并给出解决方案
```

### 5. 技术方案设计

```markdown
# Context
我们需要实现一个 [功能描述]

# Requirements
- 功能需求：[列出]
- 性能要求：[列出]
- 技术约束：[列出，如必须用 React]

# Task
请设计一个技术方案

# Output Format
1. 方案概述
2. 架构设计（包含架构图描述）
3. 核心组件/模块
4. 数据流
5. 技术选型及理由
6. 潜在风险及应对
7. 开发计划（分几个阶段）
```

## 🎯 面试高频问题

### Q1: 什么是 Prompt Engineering？为什么重要？

**参考回答**：
```
Prompt Engineering 是设计和优化提示词，让大模型输出更好结果的技术。

重要性：
1. 同样的模型，不同 prompt 效果差异巨大
2. 好的 prompt 可以激发模型的涌现能力
3. 是前端开发最常用的 AI 技能
4. 不需要训练模型，成本低见效快

实际应用：
- 代码生成和审查
- 技术文档写作
- 智能客服
- 内容创作
```

### Q2: 你用过哪些 Prompt 技巧？

**参考回答**：
```
我常用的技巧：

1. Zero-Shot：简单任务直接问
2. Few-Shot：给几个示例让模型模仿
3. Chain of Thought：让模型一步步思考，提升推理准确率
4. Role Playing：设定角色，让模型用特定视角回答
5. Output Formatting：明确要求输出格式

最常用的是 Chain of Thought，
对于复杂问题，让模型一步步思考可以显著减少错误。
```

### Q3: 写一个你实际工作中用的 Prompt 示例

**参考回答**：
```
我经常用 AI 做代码审查，这是我的 Prompt 模板：

"你是一位经验丰富的前端代码审查员。
请审查下面的 React 代码，关注：
1. 性能问题（不必要的渲染、内存泄漏）
2. 安全问题（XSS、敏感信息暴露）
3. 代码质量（可读性、类型安全）

对每个问题，指出位置、风险等级、给出改进代码。

[代码]"

这个 Prompt 明确了角色、任务、关注点和输出格式，
AI 的审查结果非常专业，帮我发现了很多潜在问题。
```
